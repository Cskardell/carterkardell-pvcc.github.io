Chapter 5: Programming Algorithms & Software Testing

Phases of Computer Program Development

  1. Phase 1: Understand the problem.
  2. Phase 2: Design an algorithm to solve the problem.
  3. Phase 3: Write the program code in an appropriate language based on the algorithm.
  4. Phase 4: Enter the program and save.
  5. Phase 5: Execute the program.
  6. Phase 6: Test the program, find errors and debug, then retest.
  7. Phase 7: Document and evaluate the program.

Programming Structures

Programming structures include:

  • Sequential structure: Top-down execution
  • Decision structures: if/then/else
  • Iterative (Looping):
    • while loop: continue looping as long something is true
    • repeat/until loop: continue looping until something is true
    • for loop: loop a given number of times.
  • Recursive structures: Repeat a set of instructions as a subtask of itself.

    Repetition continues until a degenerative case, which is a termination condition, is met (degenerative case is also called a "base case").

Two Classic Programming/Database Problems

Two classic programming/database problems include:

  1. Sorting:
    • Descending order/Ascending order.
    • Sorting numbers
    • Sorting strings (in accordance with ASCII values).
  2. Searching:

    In a database, searching can involve:

    • Filters: temporary search
    • Queries: search criteria are saved so they don't have to be recreated.

    Two popular search techniques:

    • Sequential search: search from top to bottom or bottom to top, one item at a time.
    • Binary search: only works if data is SORTED first. Start in the middle, then go to half of that, etc.

Two Classic Program Goals

Two classic program goals are:

  1. Efficiency: Program takes minimal time and/or resources.
  2. Correctness: Program runs in expected ways and works for all expected conditions.

Software Testing and Verification

Software testing is a critical aspect of software development. It involves various kinds of tests to ensure software quality:

  • Acceptance testing: Verifying whether the whole system works as intended.
  • Integration testing: Ensuring that software components or functions operate together.
  • Unit testing: Validating that each software unit performs as expected. A unit is the smallest testable component of an application.
  • Functional testing: Checking functions by emulating business scenarios, based on functional requirements. Black-box testing is a
    common way to verify functions.
  • Performance testing: Testing how the software performs under different workloads. Load testing, for example, is used to evaluate
    performance under real-life load conditions.
  • Regression testing: Checking whether new features break or degrade functionality. Sanity testing can be used to verify menus,
    functions, and commands at the surface level, when there is no time for a full regression test.
  • Stress testing: Testing how much strain the system can take before it fails. Considered to be a type of non-functional testing.
  • Usability testing: Validating how well a customer can use a system or web application to complete a task.

Software testing is done to prevent:

  • Architectural flaws
  • Poor design decisions
  • Invalid or incorrect functionality
  • Security vulnerabilities
  • Scalability issues

Testing best practices:

  • Continuous Testing: Project teams test each build as it becomes available, relying on test automation integrated with the deployment process.
  • Configuration management: Organizations centrally maintain test assets and track what software builds to test.
  • Service Visualization: Testing environments may not be available, especially early in code development.
  • Defect or Bug Tracking: Monitoring defects is important to both testing and development teams for measuring and improving quality.
  • Metrics and Reporting: Reporting and analytics enable team members to share status, goals, and test results.
